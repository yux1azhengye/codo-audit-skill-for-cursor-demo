# Round 1 专项：业务逻辑漏洞

## 目标

发现常规 SAST 难以覆盖的逻辑缺陷：参数篡改、状态机绕过、越权（IDOR）、并发条件竞争、业务流程绕过等。有效上下文为 API 的完整逻辑链路。

## 策略

1. **角色设定**：以“理解业务的高级渗透测试专家”执行本专项，打破正向开发思维，审视“用户是否可通过篡改参数/状态/顺序达到非预期结果”。
2. **敏感参数与路由**：利用变量名语义（`user_id`、`role`、`is_admin`、`skip_auth`、`amount`、`status`、`order_id`）定位敏感接口，枚举接收上述参数的路由，追踪在 Service 层的用法。
3. **状态与分支**：关注状态机（订单状态、审批流、支付状态）、标志位（`is_verified`、`paid`）、权限分支是否可被跳过或篡改。
4. **并发与竞态**：扣减、限额、库存、优惠券等是否在“读-改-写”间存在竞态窗口。
5. **注解与拦截器**：若 Recon 中标注了“路由与 Filter/拦截器/注解”的对应关系，在本专项中单独核对：标注了权限的路由是否真的经过校验，是否有路由未受保护或可被绕过。

## 上下文提取

以“路由”为单元提取完整逻辑链路（Controller → Service → 关键分支），避免全文件堆砌；优先分析 Recon 中已标记的敏感路由与业务对象。

## 输出（写入 round1-findings）

每条：**API/路由**、**业务场景简述**、**问题类型（篡改/绕过/竞态/…）**、**关键参数与数据流**。二轮补 PoC 与复现步骤。

# 误报过滤与二轮复核规则

## 一、规则过滤（二轮前必须排除）

以下类别**不得进入 confirmed-findings，也不得进入终报**：

1. **文档/非代码**：Markdown、文档、注释中的“漏洞”描述或示例代码。
2. **明确排除类型**（除非用户明确要求纳入）：
   - 纯 DoS / 资源耗尽 / 速率限制
   - 内存/CPU 理论安全问题且**无当前代码中的可触发证据**
   - 开放重定向且**无内网/敏感协议利用链**（可记附录）
3. **无证据链**：无清晰 Source→Sink、或无法给出可复现 PoC 的条目 → 仅可记入“待确认”，不进入终报。

## 二、AI 复核指令（对通过规则过滤的条目）

对每条 Round 1 发现逐条确认：

1. **数据流完整性**：Source 是否真实可控（来自请求/配置/可影响的外部输入）？到 Sink 的路径是否无断点？
2. **可触发性**：是否存在有效防护（WAF、参数化、白名单、鉴权）？若存在且不可绕过，则降级为“已防护”或剔除。
3. **证据充分性**：是否有代码片段或调用链支撑？若仅为推测，标记为“待确认”或剔除。

## 三、组织级误报清单（可选）

若项目或组织维护了「视为误报」的清单（如特定依赖的已知误报、历史已修复项），应将该清单内容注入本阶段 Prompt，在 AI 复核时排除这些项，避免重复报出。

## 四、输出

- 通过复核的条目 → 写入 **confirmed-findings.md**，作为终报唯一数据源。
- 被剔除的条目可简要记录原因（规则过滤 / 已防护 / 无 PoC），便于审计追溯。
